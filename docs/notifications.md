# In-App Notification System

## Overview

The in-app notification system delivers safety-conscious alerts to authenticated users within the platform. Every notification is stored persistently in the database and, when the recipient has a live socket connection, also pushed in real time without requiring a page refresh or polling cycle.

Notification wording follows a **discreet copy convention** (SSD §22.2): messages are deliberately vague ("You have a new update.", "A report you submitted has been updated.") so that notification surfaces (browser push, notification banners) do not inadvertently reveal sensitive context to bystanders.

---

## Architecture

### Layers

```
Write path (backend)
  └── notificationService.js  ← single write path for all notification types
        ├── InAppNotification.create()   (MySQL row)
        └── io.to('user:<userId>').emit('notification:new')  (real-time push)

Read path (frontend polling)
  └── NotificationBell (SiteHeader)
        └── GET /api/notifications  (every 30 seconds)

Real-time path (frontend socket)
  └── notificationSocket singleton
        └── 'notification:new' event
              └── NotificationBell handler → badge increment
```

The DB write **always** happens first. The socket push is a best-effort layer on top — if `io` is not yet available (e.g. during unit tests or very early bootstrap), the write succeeds and the 30-second poll catches the notification on the next cycle.

## Backend Service (`backend/src/services/notificationService.js`)

### Initialisation

`setNotificationIo(io)` must be called once at server startup (after the Socket.io server is created) to wire up real-time push. This is done in `backend/index.js`:

```js
const { setNotificationIo } = require('./src/services/notificationService');
setNotificationIo(io);
```

If this is not called, a `console.warn` is emitted per notification and push is skipped — polling still works.

### `createNotification(opts)`

Persists a single notification and pushes a `notification:new` event to the recipient's personal socket room.

**Parameters:**

| Field | Type | Required | Description |
|---|---|---|---|
| `recipientUserId` | `string` | Yes | `UserAccount.userId` of the recipient. |
| `message` | `string` | Yes | Discreet notification text (≤255 chars). Must not expose sensitive context. |
| `category` | `string` | No | `notificationCategoryType` ENUM value. Default: `'REPORT_UPDATE'`. |
| `entityType` | `string\|null` | No | `relatedEntityType` — e.g. `'CHAT'`, `'REPORT'`. Enables frontend deep-linking. |
| `entityId` | `string\|null` | No | `relatedEntityId` — UUID of the referenced entity. |

**Returns:** `Promise<InAppNotification>` — the persisted Sequelize row.

**Side effects:**
- Writes one row to `InAppNotification`.
- Emits `notification:new` to `user:<recipientUserId>` via Socket.io (if `io` is set).

### `createNotificationsBulk(recipientUserIds, message, category, entityType, entityId)`

Fan-out helper. Calls `createNotification` in parallel for each recipient. Failures for individual recipients are swallowed (logged to `console.error`) so one bad `userId` does not block the rest of the batch.

**Callers:**

| Caller | Trigger | Category | Entity |
|---|---|---|---|
| `chatSocket.js` — `createDiscreetNotifications` | New direct-chat message | `'NEW_MESSAGE'` | `'CHAT'`, `channel.chatId` |
| `reportController.js` | Report status change | `'REPORT_UPDATE'` | `'REPORT'`, `reportId` |
| `communityController.js` | Moderation warning issued | `'MODERATION_ACTION'` | _(none)_ |

## Data Model (`InAppNotification`)

| Column | Type | Description |
|---|---|---|
| `notificationId` | `VARCHAR(36) PK` | UUID generated by `randomUUID()`. |
| `recipientUserId` | `VARCHAR(36) FK` | `UserAccount.userId`. |
| `notificationCategoryType` | ENUM | `'REPORT_UPDATE'`, `'NEW_MESSAGE'`, `'MODERATION_ACTION'`, `'SYSTEM'` |
| `discreetNotificationMessage` | `VARCHAR(255)` | The vague notification text shown to the user. |
| `notificationReadStatus` | ENUM | `'UNREAD'` on creation; `'READ'` after `PATCH /api/notifications/:id/read`. |
| `notificationDismissedStatus` | ENUM | `'VISIBLE'` on creation; `'DISMISSED'` after dismiss. Separate from read so "mark read" and "dismiss" are independent actions. |
| `notificationCreationTimestamp` | `DATETIME` | Set at creation time. |
| `relatedEntityType` | `VARCHAR(50)\|NULL` | Type string for deep-linking (`'CHAT'`, `'REPORT'`, etc.). |
| `relatedEntityId` | `VARCHAR(36)\|NULL` | UUID of the related entity. |

## REST API

All notification endpoints require a valid JWT (`Authorization: Bearer <token>`).

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/notifications` | Returns all non-dismissed notifications for the authenticated user, ordered by creation time (newest first). |
| `PATCH` | `/api/notifications/:notificationId/read` | Marks a single notification as `READ`. |
| `PATCH` | `/api/notifications/:notificationId/dismiss` | Marks a notification as `DISMISSED`. Dismissed notifications are excluded from subsequent `GET` responses. |
| `PATCH` | `/api/notifications/mark-all-read` | Marks every UNREAD notification for the current user as READ. |

## Real-Time Push (Socket.io)

When `createNotification` writes a row, it immediately emits to `io.to('user:<recipientUserId>')`. The recipient's socket joins this personal room at `chatSocket.js` connection time — so any authenticated user who has a chat socket open will receive real-time notification push without extra subscription steps.

### `notification:new` event payload

```json
{
  "notificationId": "uuid-v4",
  "category": "NEW_MESSAGE",
  "message": "You have a new update.",
  "createdAt": "2026-06-30T10:00:00.000Z",
  "entityType": "CHAT",
  "entityId": "chat-channel-uuid"
}
```

## Frontend Integration

### NotificationBell (`frontend/src/components/NotificationBell.jsx`)

The bell icon in `SiteHeader` is the primary notification surface. It:

1. Polls `GET /api/notifications` every **30 seconds** for new/unread counts and message content.
2. Subscribes to `notificationSocket` for real-time `notification:new` events.
3. Shows an unread badge count.
4. Exposes a dropdown panel listing recent notifications with dismiss and mark-read actions.

The 30-second poll is the **reconciliation fallback** — it catches notifications that arrived during a socket disconnection window. The socket push provides the **zero-latency path** for users with an active connection.

### notificationSocket (`frontend/src/services/notificationSocket.js`)

A module-level singleton socket.io-client. It uses `autoConnect: false` so the connection is deferred until the user is authenticated.

```js
// App.jsx — connect on sign-in
import notificationSocket from './services/notificationSocket';
notificationSocket.connect(authToken);

// NotificationBell — subscribe to push
notificationSocket.subscribe((payload) => {
  setUnreadCount(prev => prev + 1);
  setNotifications(prev => [payload, ...prev]);
});

// App.jsx — disconnect on sign-out
notificationSocket.disconnect();
```

Multiple components can call `subscribe`/`unsubscribe` without opening duplicate connections because the underlying socket.io instance is shared.

## Discreet Wording Convention

All notification messages must follow the discreet copy standard (SSD §22.2). The rationale is that notification surfaces (push banners, OS notification trays, lock screens) may be visible to bystanders. Exposing that a survivor has received a message from a "counsellor" or that a "GBV report" has been updated could endanger them.

**Approved patterns:**

| Trigger | Message |
|---|---|
| New direct-chat message | `"You have a new update."` |
| Report status changed | `"A report you submitted has been updated."` |
| Moderation warning | `"An action has been taken on your account."` |

**Never include:** names, roles, case numbers, report IDs, or anything that reveals the user's platform context.

The `entityType` and `entityId` fields may still be set — they are used by the frontend to deep-link the notification to the correct page after the user opens the platform. These fields are not shown in the notification text itself.

## Notification Lifecycle

```
createNotification() called
        │
        ▼
InAppNotification row created
notificationReadStatus = UNREAD
notificationDismissedStatus = VISIBLE
        │
        ├── Socket push → notification:new (if recipient connected)
        │
        ▼
GET /api/notifications → appears in bell dropdown
        │
        ├── PATCH …/read    → notificationReadStatus = READ   (badge decrements)
        └── PATCH …/dismiss → notificationDismissedStatus = DISMISSED (hidden from list)
```

Dismiss and read are independent so the frontend can let users clear the dropdown without affecting the read state, or mark something read without dismissing it.
