# Data Model Reference

This document describes every Sequelize model in `backend/src/models/`, the associations between them, and the architectural decisions that shape the schema. It is intended as a reliable reference for developers who are new to this codebase.

---

## Architectural Overview

### Single Registry Pattern

All models are imported and all associations are defined in **`backend/src/models/index.js`**. Every controller, service, and socket handler must import models from this file:

```js
const { UserAccount, SurvivorProfile, IncidentReport } = require('../models');
```

Direct imports from individual model files (e.g. `require('./userAccount')`) skip all association definitions and break eager-loading. Never do this outside of `models/index.js` itself.

### Identity Architecture

`userAccount` is the root identity table. Every platform user — regardless of role — has exactly one row in `userAccount`. Role-specific attributes live in separate profile tables that extend it via a `userId` foreign key:

```
userAccount (root)
  ├── survivorProfile       (role: SURVIVOR)
  ├── counsellorProfile     (role: COUNSELLOR)
  ├── legalCounselProfile   (role: LEGAL_COUNSEL)
  ├── ngoAdministratorProfile (role: NGO_ADMIN)
  └── moderatorProfile      (role: MODERATOR)
```

This design means:
- Auth middleware only needs to check `userAccount` (one fast PK lookup)
- Role-specific data is cleanly separated — a counsellor's `currentWorkloadScore` cannot pollute the survivor table
- Cascade deletes on `userId` propagate correctly

### UUID Primary Keys

Most domain entities use `VARCHAR(36)` UUIDs as primary keys (32 hex digits + 4 hyphens). Generated by `crypto.randomUUID()` in the application layer before insert. Rationale: auto-increment integer IDs are enumerable, which means an attacker can guess adjacent resource IDs. UUID PKs prevent this class of enumeration attack.

The one exception is `systemSetting`, which uses a human-readable `settingKey` as its PK for simpler upsert patterns.

### MySQL BOOLEAN = TINYINT(1)

`DataTypes.BOOLEAN` in Sequelize maps to `TINYINT(1)` in MySQL. MySQL has no native boolean type. This is a MySQL rendering detail, not a schema mistake. When reading raw SQL output you will see `0` and `1` for boolean columns.

### ENUM Drift Handling

MySQL ENUMs cannot be altered without an `ALTER TABLE`, which is a schema change that must not break existing rows with values that have not changed. This project handles ENUM additions automatically via `backend/src/utils/schemaCompatibility.js`, which runs idempotent DDL guards on every server boot. **Do not run manual `ALTER TABLE` commands** — add reconciliation steps to `schemaCompatibility.js` instead. Set `ENABLE_SCHEMA_COMPAT=false` to disable temporarily.

## Detailed Model Reference

### UserAccount

**File:** `backend/src/models/userAccount.js`  
**Purpose:** Base identity table for every platform user. Authentication state, OTP fields, ECDH public key, and account lifecycle status all live here.

| Column | Type | Notes |
|---|---|---|
| `userId` | VARCHAR(36) PK | UUID generated by `crypto.randomUUID()` before insert |
| `phoneNumber` | VARCHAR(15) UNIQUE NOT NULL | Primary login identifier |
| `hashedPassword` | VARCHAR(255) | bcrypt hash — never stored as plaintext |
| `userRole` | ENUM | `SURVIVOR`, `COUNSELLOR`, `LEGAL_COUNSEL`, `NGO_ADMIN`, `MODERATOR` |
| `role` | STRING | Lowercase compatibility alias for `userRole` (legacy frontend use) |
| `status` | STRING | Compatibility field; `accountStatus` is authoritative |
| `accountStatus` | ENUM NOT NULL | `ACTIVE`, `SUSPENDED`, `DEACTIVATED`, `BANNED` |
| `banReason` | TEXT | Human-readable reason entered by NGO admin at ban time |
| `bannedAt` | DATE | UTC timestamp when the ban was applied |
| `banExpiresAt` | DATE | NULL = permanent ban; set = auto-lifted at next auth check |
| `bannedByUserId` | VARCHAR(36) | userId of the NGO admin who issued the ban (not a FK) |
| `isOtpVerified` | BOOLEAN | TRUE after successful OTP verification at registration |
| `accountCreationTimestamp` | DATE | UTC timestamp of account creation |
| `otpHash` | VARCHAR(255) | bcrypt hash of the active OTP (any purpose); plaintext never stored |
| `otpPurpose` | VARCHAR(40) | `SIGNUP_OTP`, `SIGNIN_OTP`, `PASSWORD_RESET`, `SIGNUP_TICKET` |
| `otpExpiresAt` | DATE | Expiry of the current OTP |
| `otpAttemptCount` | INTEGER UNSIGNED | Failed OTP verification attempts for the current OTP |
| `authFailedAttempts` | INTEGER UNSIGNED | Consecutive failed password login attempts |
| `authLockUntil` | DATE | Temporary lockout timestamp after repeated auth failures |
| `ecdhPublicKey` | LONGTEXT | ECDH P-256 public key (JWK string) for E2EE direct chat |

**Associations:** `hasOne` each profile table; `hasMany` DirectChatChannel, DirectChatMessage, CommunityMessage, HarmfulContentReport, ModerationActionLog (×2, as `moderationActions` and `receivedModerationActions`), AuditLog, InAppNotification, SupportResource, ResourceAccessEvent, RoomMembership, StaffReassignmentRequest.

### SurvivorProfile

**File:** `backend/src/models/survivorProfile.js`  
**Purpose:** Survivor-specific attributes. Every SURVIVOR user has exactly one row. Holds both the public pseudonym (`displayNickname`) and private attributes (`assignedGender`, `residenceCounty`) visible only to assigned staff.

| Column | Type | Notes |
|---|---|---|
| `survivorId` | VARCHAR(36) PK | UUID — separate from `userId` for profile isolation |
| `userId` | VARCHAR(36) UNIQUE NOT NULL | FK to `userAccount` |
| `displayNickname` | VARCHAR(50) NOT NULL | Public pseudonym in community rooms — not a real name |
| `assignedGender` | VARCHAR(20) | Private — visible only to survivor and assigned staff |
| `residenceCounty` | VARCHAR(50) NOT NULL | Used for geographic counsellor auto-assignment |
| `assignedCounsellorId` | VARCHAR(36) | FK to `counsellorProfile` — set by auto-assignment at registration |
| `assignedLegalCounselId` | VARCHAR(36) | FK to `legalCounselProfile` — set by auto-assignment at registration |
| `privacyPreferencesJson` | JSON | Optional blob for notification opt-outs and display settings |

**Associations:** `belongsTo` UserAccount; `belongsTo` CounsellorProfile (as `assignedCounsellor`, `onDelete: SET NULL`); `belongsTo` LegalCounselProfile (as `assignedLegalCounsel`, `onDelete: SET NULL`); `hasMany` IncidentReport, DirectChatChannel, StaffAssignmentHistory, StaffReassignmentRequest.

### IncidentReport

**File:** `backend/src/models/incidentReport.js`  
**Purpose:** The central reporting record. A survivor submits an incident report that progresses through a 7-state lifecycle as NGO staff review, support, investigate, and resolve it.

**State Machine:**
```
SUBMITTED → UNDER_REVIEW → ACTIVE_SUPPORT → UNDER_INVESTIGATION
         → LEGAL_REVIEW → ESCALATED_TO_LEGAL_CASE
         → RESOLVED
         → WITHDRAWN
```

| Column | Type | Notes |
|---|---|---|
| `reportId` | VARCHAR(36) PK | UUID |
| `survivorId` | VARCHAR(36) NOT NULL | FK to `survivorProfile` (CASCADE on delete) |
| `incidentCategory` | VARCHAR(50) NOT NULL | GBV category — `physical_violence`, `sexual_violence`, etc. |
| `severityLevel` | VARCHAR(20) NOT NULL | Triage level: `LOW`, `MEDIUM`, `HIGH`, `CRITICAL` |
| `incidentDescriptionText` | TEXT NOT NULL | Full survivor narrative |
| `incidentLocation` | VARCHAR(100) | Location of the incident (not the survivor's address) |
| `incidentDate` | DATEONLY | Date the incident occurred |
| `currentReportStatus` | ENUM NOT NULL DEFAULT 'SUBMITTED' | Position in the lifecycle state machine |
| `reportCreationTimestamp` | DATE | UTC creation timestamp |

**Associations:** `belongsTo` SurvivorProfile; `hasMany` EvidenceFile (CASCADE); `hasOne` LegalCaseFile (CASCADE, UNIQUE on `reportId` enforces 1:0..1).

### DirectChatChannel

### DirectChatMessage

**File:** `backend/src/models/directChatMessage.js`  
**Purpose:** Individual message within a `DirectChatChannel`. Content is always stored encrypted — the server never holds plaintext. The ECDH P-256 + AES-GCM key agreement happens on the client side; only ciphertext is transmitted and persisted.

| Column | Type | Notes |
|---|---|---|
| `messageId` | VARCHAR(36) PK | UUID |
| `chatId` | VARCHAR(36) NOT NULL | FK to `directChatChannel` (CASCADE) |
| `senderUserId` | VARCHAR(36) NOT NULL | FK to `userAccount` (as `sender`) |
| `encryptedMessageContent` | TEXT NOT NULL | E2EE ciphertext — never decrypted server-side |
| `messageDispatchTimestamp` | DATE | UTC timestamp of send |
| `messageReadStatus` | ENUM DEFAULT 'UNREAD' | `UNREAD`, `READ` |
| `deliveredAt` | DATE | UTC timestamp when message reached an online recipient (NULL = not yet delivered) |
| `seenAt` | DATE | UTC timestamp when the recipient read the message (NULL = not yet seen) |

**Associations:** `belongsTo` DirectChatChannel; `belongsTo` UserAccount (as `sender`).

### SystemSetting

**File:** `backend/src/models/systemSetting.js`  
**Purpose:** Key/value store for durable platform-level settings that must survive process restarts. Currently used only for maintenance mode, but designed to hold any JSON-serialised platform configuration.

| Column | Type | Notes |
|---|---|---|
| `settingKey` | VARCHAR(64) PK | Well-known identifier (e.g. `'maintenance'`) |
| `settingValue` | TEXT | JSON-serialised value — callers must `JSON.parse` on read |
| `createdAt` | DATE | Auto-managed by Sequelize `timestamps: true` |
| `updatedAt` | DATE | Auto-managed by Sequelize `timestamps: true` |

Known keys:
- `'maintenance'` — JSON object `{ enabled: boolean, message?: string }` — persisted when NGO Admin toggles maintenance mode. Loaded into an in-process cache at boot via `loadMaintenanceStateFromDb()` so the maintenance guard never needs a DB round-trip per request.

## Import Pattern for Controllers

All controllers must import from the central registry:

```js
// Correct
const { UserAccount, SurvivorProfile, IncidentReport } = require('../models');

// Wrong — skips all associations
const UserAccount = require('../models/userAccount');
```

The registry exports `sequelize` directly so you can run raw queries if needed:

```js
const { sequelize } = require('../models');
const [rows] = await sequelize.query('SELECT COUNT(*) as total FROM incidentReport');
```
