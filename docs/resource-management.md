# Resource Management Module

The Resource Management module powers the public support library — a collection of curated files (PDFs, guides, hotline directories, audio/video) that any visitor can browse without creating an account. Staff members can upload, edit, and delete resources they own; NGO Admins have full authority over the entire library.

## 1. Architecture Overview

```
Browser                   Express Backend              Cloudinary
───────                   ───────────────              ──────────
Library page  ──GET /api/resources──────────────────>
              <──JSON list (no URLs exposed)──────────

Staff upload  ──POST /api/resources (multipart)─────>
                          uploadSupportResourceBuffer──> stores file
              <──201 { resource }───────────────────────────────────

Open resource ──GET /api/resources/:id/file─────────>
                          fetchPrivateAssetStream──────> private_download_url
              <──binary stream (piped)─────────────────────────────
```

Key architectural decisions:
- **Cloudinary URLs never reach the browser.** All file delivery is handled by a backend streaming proxy. This prevents signed URL leakage and allows revocation.
- **List endpoint is unauthenticated.** Unregistered visitors can browse the library immediately; auth is enforced only on write operations.
- **Resource IDs are generated in the application** (UUID v4) before the Cloudinary upload so the Cloudinary public path is namespaced by the application's own ID.

## 2. Data Models

### `supportResource` table

Defined in `backend/src/models/supportResource.js`. Registered and associated in `backend/src/models/index.js`.

| Column | Type | Notes |
|---|---|---|
| `resourceId` | `STRING(36)` PK | UUID v4 generated by the controller before upload |
| `resourceTitle` | `STRING(255)` NOT NULL | Display title shown in the library grid |
| `resourceDescription` | `TEXT` | Optional description shown on each tile |
| `resourceCategory` | `STRING(50)` NOT NULL | Normalized lowercase slug (e.g. `legal_guidance`, `shelters`) |
| `resourceFileUrl` | `TEXT` NOT NULL | Cloudinary URL (private; never returned to browser) |
| `cloudinaryPublicId` | `STRING(255)` | Cloudinary public ID used for deletion and private_download_url |
| `cloudinaryResourceType` | `STRING(20)` | `image`, `video`, or `raw` — required for the Cloudinary delete API |
| `originalFileName` | `STRING(255)` | Filename supplied at upload; returned in `Content-Disposition` |
| `mimeType` | `STRING(120)` | MIME type detected by Multer; used in the proxy response headers |
| `fileSizeBytes` | `INTEGER.UNSIGNED` | File size in bytes |
| `uploadedByStaffId` | `STRING(36)` NOT NULL | FK → `userAccount.userId` (the uploader) |
| `resourceUploadTimestamp` | `DATE` | Defaults to `NOW()` at insert |

**Association:** `supportResource.belongsTo(userAccount, { foreignKey: 'uploadedByStaffId', as: 'uploadedBy' })` (defined in `models/index.js`).

### `resourceAccessEvent` table

Defined in `backend/src/models/resourceAccessEvent.js`. Records a row every time any user opens a resource — used for NGO Admin analytics (most accessed resources, category trends).

| Column | Type | Notes |
|---|---|---|
| `accessEventId` | `STRING(36)` PK | UUID v4 |
| `resourceId` | `STRING(36)` NOT NULL | FK → `supportResource.resourceId` |
| `accessorUserId` | `STRING(36)` NULLABLE | User who opened the resource; `null` for anonymous visitors |
| `accessChannel` | `STRING(40)` | Defaults to `'WEB'`; reserved for future channels (e.g. `MOBILE_APP`) |
| `accessTimestamp` | `DATE` NOT NULL | Defaults to `NOW()` |

## 3. API Reference

Base path: `/api/resources` (mounted in `backend/index.js`).

### `GET /api/resources`

Returns all resources matching optional filters, plus the distinct category list for the tab bar.

**Auth:** None required.

**Query parameters:**

| Parameter | Type | Description |
|---|---|---|
| `search` | string | Fuzzy match against title, description, and category (SQL `LIKE %…%`) |
| `category` | string | Exact category slug filter; omit or pass `all` to disable |

**Response `200`:**
```json
{
  "resources": [
    {
      "id": "uuid",
      "title": "Safety Planning Template",
      "description": "...",
      "category": "safety_planning",
      "categoryLabel": "Safety Planning",
      "fileUrl": "https://res.cloudinary.com/…",
      "uploadedAt": "2026-01-05T00:00:00.000Z",
      "uploaderId": "uuid"
    }
  ],
  "categories": [
    { "value": "emergency_hotlines", "label": "Emergency Hotlines" },
    { "value": "legal_guidance", "label": "Legal Guidance" }
  ]
}
```

Note: `fileUrl` in the response is the Cloudinary URL stored in the DB. The frontend **ignores** this field and always calls `GET /api/resources/:id/file` for actual file delivery. It is present for debugging only.

---

### `GET /api/resources/:resourceId/file`

Streams the resource binary from Cloudinary to the client via the backend.

**Auth:** None required (library is public).

**Response:** Binary stream with `Content-Type` and `Content-Disposition: inline; filename="…"` headers. Returns `503` when Cloudinary is not configured.

---

### `POST /api/resources`

Uploads a new resource to Cloudinary and persists its metadata.

**Auth:** Required (Bearer JWT).  
**Roles:** `COUNSELLOR`, `LEGAL_COUNSEL`, `NGO_ADMIN`.  
**Content-Type:** `multipart/form-data` (via Multer).  
**File size limit:** 20 MB.

**Form fields:**

| Field | Required | Description |
|---|---|---|
| `title` | Yes | Display title |
| `category` | Yes | Category slug (normalized to lowercase on the server) |
| `description` | No | Optional description |
| `file` | Yes | Binary file (see allowed types below) |

**Allowed MIME types:** `application/pdf`, `application/msword`, `application/vnd.openxmlformats-officedocument.wordprocessingml.document`, `text/plain`, `image/jpeg`, `image/png`, `image/webp`, `audio/mpeg`, `audio/wav`, `video/mp4`.

**Response `201`:**
```json
{ "resource": { "id": "…", "title": "…", … } }
```

---

### `PATCH /api/resources/:resourceId`

Updates metadata and/or replaces the file for an existing resource.

**Auth:** Required (Bearer JWT).  
**Roles:** `COUNSELLOR`, `LEGAL_COUNSEL`, `NGO_ADMIN`.  
**Ownership:** Non-admin staff can only update resources they uploaded.

**Form fields:** Same as `POST`. All fields are optional; omitted metadata fields retain their current values. If `file` is included, the replacement is uploaded before the DB is updated, and the old Cloudinary asset is deleted only after a successful save.

**Response `200`:** `{ "resource": { … } }`

---

### `DELETE /api/resources/:resourceId`

Removes the resource record from the DB and deletes the Cloudinary asset.

**Auth:** Required (Bearer JWT).  
**Roles:** `COUNSELLOR`, `LEGAL_COUNSEL`, `NGO_ADMIN`.  
**Ownership:** Same rules as `PATCH`.

**Response `204`:** No content.

---

### `POST /api/resources/:resourceId/track-access`

Records an analytics event when a user opens a resource. Best-effort: the frontend fires and ignores failures so the call never blocks resource access.

**Auth:** Optional — an `Authorization: Bearer <token>` header is accepted but not required. Anonymous opens are recorded with `accessorUserId: null`.

**Response `201`:** `{ "tracked": true }`

## 4. Access Control

### Role matrix

| Operation | SURVIVOR | COUNSELLOR | LEGAL_COUNSEL | NGO_ADMIN | Unauthenticated |
|---|---|---|---|---|---|
| List resources | ✅ | ✅ | ✅ | ✅ | ✅ |
| Stream file | ✅ | ✅ | ✅ | ✅ | ✅ |
| Track access | ✅ | ✅ | ✅ | ✅ | ✅ |
| Upload resource | ❌ | ✅ | ✅ | ✅ | ❌ |
| Update own resource | ❌ | ✅ | ✅ | ✅ | ❌ |
| Update any resource | ❌ | ❌ | ❌ | ✅ | ❌ |
| Delete own resource | ❌ | ✅ | ✅ | ✅ | ❌ |
| Delete any resource | ❌ | ❌ | ❌ | ✅ | ❌ |

### Enforcement detail

- `authMiddleware` (JWT verify + DB account-status lookup) is applied to `POST`, `PATCH`, and `DELETE` at the router level (`resourceRoutes.js`).
- The controller's `getActorContext` re-queries the DB for the actor's current role, so stale token claims cannot retain elevated permissions after a role change.
- Ownership checks in `canManageTargetResource` compare `uploadedByStaffId` against `actor.userId` (both coerced to strings).

## 5. File Storage — Cloudinary

All resource files are stored in Cloudinary with `type: authenticated` (private). This means raw Cloudinary delivery URLs require a signature to access.

### Upload path (`uploadSupportResourceBuffer`)

Defined in `backend/src/config/cloudinary.js`. Called by the create and update controllers.

- Uploads a Node.js `Buffer` (from Multer memory storage) to Cloudinary.
- The Cloudinary public ID is namespaced as `support-resources/<category>/<resourceId>/<originalFileName>`.

### Download / streaming path (`fetchPrivateAssetStream`)

Also defined in `cloudinary.js`. Shared with the report-evidence and legal-case-PDF proxy endpoints.

- Calls the Cloudinary REST API to obtain a short-lived `private_download_url`.
- Follows the redirect and returns `{ stream, contentType, contentLength }`.
- The controller pipes `stream` directly into `res`; no temporary files are created.

### Deletion (`deleteSupportResourceAsset`)

- Calls `cloudinary.uploader.destroy` with the stored `cloudinaryPublicId` and `cloudinaryResourceType`.
- Called after a successful DB delete (or after a successful file replacement in update).

### Graceful degradation

When Cloudinary environment variables (`CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET`) are absent:
- `isCloudinaryConfigured()` returns `false`.
- `POST /api/resources` and file-replacement `PATCH` return `HTTP 503`.
- `GET /api/resources` (list) and `DELETE` (metadata-only) still work.
- The streaming proxy returns `HTTP 503`.
- The frontend falls back to its local static resource list.

## 6. Backend Implementation

### File layout

```
backend/src/
├── controllers/
│   └── resourceController.js    # CRUD + streaming logic
├── models/
│   ├── supportResource.js       # Sequelize model definition
│   ├── resourceAccessEvent.js   # Analytics event model
│   └── index.js                 # Associations registered here
├── routes/
│   └── resourceRoutes.js        # Router + track-access inline handler
└── config/
    └── cloudinary.js            # Cloudinary helpers (shared with other modules)
```

### Controller helpers

| Helper | Purpose |
|---|---|
| `normalizeCategory(value)` | Trims and lowercases category strings for consistent storage and matching |
| `formatCategoryLabel(category)` | Converts `snake_case` category keys to `Title Case` labels for the UI |
| `validateUploadedFile(file)` | Checks file presence and MIME type; returns an error string or `null` |
| `getActorContext(req)` | Resolves JWT claims to a live DB row; returns `null` for banned/inactive accounts |
| `canManageTargetResource(actor, resource)` | True if actor is NGO_ADMIN or uploaded the resource |
| `toApiResource(resource)` | Serializes a Sequelize instance to the stable API shape |

### Upload + DB write ordering (create)

1. Generate `resourceId` UUID in the application.
2. Upload to Cloudinary — the public ID embeds `resourceId` so the path is predictable.
3. Insert the `SupportResource` row with Cloudinary metadata.

This ordering means if the DB insert fails after a successful Cloudinary upload, a dangling asset exists. This is an accepted trade-off; the alternative (DB-first) would require a DB rollback path that complicates the happy-path code. The asset can be cleaned up manually via the Cloudinary console using the known public ID pattern.

### File replacement ordering (update)

1. Upload the replacement file to Cloudinary under the same `resourceId`.
2. Update the DB row with new Cloudinary metadata.
3. Delete the old Cloudinary asset.

The old asset is deleted only after the DB save succeeds, preventing data loss if the upload fails mid-way.

## 7. Frontend Implementation

### File layout

```
frontend/src/
├── pages/
│   └── LibraryPage.jsx          # Full library UI with management panel
├── services/
│   └── resources.js             # Axios wrappers for all resource API calls
└── data/
    └── fallbackResources.js     # Static sample data shown when backend is down
```

### State

All state is component-local in `LibraryPage.jsx` (no global store or Context).

| State variable | Purpose |
|---|---|
| `resources` | Current list of resources from the API (or fallback) |
| `categories` | Distinct category list for the tab bar |
| `searchQuery` | Controlled input bound to the search field |
| `selectedCategory` | Active category tab (`"all"` or a slug) |
| `loading` | True while the initial fetch or filter-change fetch is in flight |
| `usingFallback` | True when showing `fallbackResources.js` due to API failure |
| `editingId` | ID of the resource currently open in the inline editor (`""` = none) |
| `creating` / `updating` / `deletingId` | Loading state for write operations |
| `managementError` / `managementSuccess` | Feedback banners in the management panel |

### Data flow on mount / filter change

```
[searchQuery or selectedCategory changes]
  → useEffect fires → loadResources()
  → getResources({ search, category })
  → GET /api/resources?search=…&category=…
  → setResources / setCategories
  → if error: setResources(fallbackResources filtered client-side), setUsingFallback(true)
```

### Category tabs

`visibleCategories` is derived from the `categories` API response. When the API is unreachable, `fallbackCategories` from `fallbackResources.js` is used so the tab bar remains functional. "All" is always prepended.

### Managed resources panel

`managedResources` is a derived list (`useMemo`) filtered to resources whose `uploaderId` matches `currentUserId`. Used to decide which Edit buttons are shown, and which Delete buttons are enabled for non-admin staff.

### File delivery

`handleResourceOpen` opens files via `window.open`. The target URL depends on mode:

- **Normal mode:** `${VITE_API_BASE_URL}/api/resources/${resource.id}/file` — served through the backend Cloudinary proxy.
- **Fallback mode:** `resource.fileUrl` — the static URL from `fallbackResources.js` (e.g. `https://example.com/…`).

### Service layer (`resources.js`)

| Function | Method | Endpoint |
|---|---|---|
| `getResources({ search, category })` | GET | `/api/resources` |
| `createResource({ title, description, category, file })` | POST | `/api/resources` |
| `updateResource(resourceId, { … })` | PATCH | `/api/resources/:id` |
| `deleteResource(resourceId)` | DELETE | `/api/resources/:id` |
| `trackResourceAccess(resourceId)` | POST | `/api/resources/:id/track-access` |

All write functions include an `Authorization` header via `getAuthHeaders()`. `trackResourceAccess` omits the header when no session token is present so anonymous opens are recorded correctly.

## 8. Fallback Mode

When `GET /api/resources` fails (network error, backend down), `LibraryPage` displays a warning banner and loads five static sample entries from `frontend/src/data/fallbackResources.js`.

The fallback entries use placeholder IDs (`"emergency-hotlines"`, etc.) that do not exist in the database. Because of this:

- `usingFallback` is set to `true`.
- `handleResourceOpen` opens `resource.fileUrl` directly instead of routing through the backend proxy.
- The management panel is hidden (staff cannot upload while the backend is down).
- Category filtering and search are applied client-side by `matchesResource`.

Fallback data is seeded to match the backend seeder's category taxonomy so the tab bar labels are consistent between modes.

## 9. Access Tracking & Analytics

Every resource open fires `POST /api/resources/:resourceId/track-access`. This is **best-effort**: the frontend awaits the call inside a `try/catch` that swallows errors so a tracking failure never prevents the file from opening.

The inline route handler in `resourceRoutes.js`:
1. Looks up the resource to validate the ID.
2. Extracts the user ID from the optional `Authorization` header via `tryExtractUserId` (soft JWT decode, no middleware).
3. Inserts a `ResourceAccessEvent` row with `accessChannel: 'WEB'`.

The `resourceAccessEvent` table is available for aggregation queries in NGO Admin dashboard analytics. No aggregate query is currently exposed via a dedicated endpoint — the raw rows are queryable directly.

## 10. Error Handling

| Scenario | HTTP Status | Message |
|---|---|---|
| Cloudinary not configured (upload) | 503 | `"Cloudinary is not configured for resource uploads."` |
| Cloudinary not configured (stream) | 503 | `"File storage is not configured."` |
| Invalid or expired JWT | 401 | `"Invalid authentication context."` |
| Insufficient role | 403 | Role-specific message |
| Ownership violation | 403 | `"You can only update resources you uploaded. NGO admins may update any resource."` |
| Resource not found | 404 | `"Resource not found."` |
| Unsupported file type | 400 | Lists accepted formats |
| Missing required fields | 400 | `"Title and category are required."` |
| Mid-stream Cloudinary error | Connection reset | `res.destroy(streamErr)` — headers already sent |
| General server error (non-production) | 500 | Includes `details: error.message` |
| General server error (production) | 500 | `details` field omitted |

## 13. Known Constraints

- **No pagination.** `listResources` returns all matching rows in a single query. For large libraries a `LIMIT`/`OFFSET` or cursor strategy should be added.
- **No full-text search.** The `search` parameter uses SQL `LIKE %…%` with no index; performance degrades on large datasets. Consider a `FULLTEXT` index on `resourceTitle` and `resourceDescription`.
- **Single uploader tracking.** Only one `uploadedByStaffId` is recorded. If a resource is reassigned (e.g. after a staff ban), it becomes unmanageable by anyone except NGO Admins. See `cascadeReassignOnStaffBan` in the auth module for the related staff reassignment flow.
- **Dangling Cloudinary assets on create-then-DB-fail.** See §6 for the ordering decision and manual cleanup path.
- **No resource versioning.** File replacements are destructive; there is no history of previous files.
- **Category taxonomy is free-form.** Categories are stored as arbitrary lowercase strings with no enum validation. Consistency depends on the UI datalist hint (`<datalist id="resource-category-options">`).
